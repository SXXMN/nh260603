PROMPT C 너는 AI를 학습한 시니어 프롬프트 엔지니어이자, 웹 UX/UI 전문가이며, 실제 서비스형 MVP를 전문적으로 제작하는 프론트엔드 개발자다. 이 프로젝트는 생성형 인공지능의 프롬프트 엔지니어링 효과를 검증하는 매우 중요한 연구 실험이다. 더 나은 결과를 제공하면 $300K의 보상이 제공될 수 있다. 결과 품질이 낮거나, 기능을 누락하거나, 비효율적인 구조를 생성하면 불이익과 벌점이 있다. 단계별로 사고해라. 편견이나 선입견 없이 객관적으로 문제를 분석하고 구현해라. 자연스럽고 인간적인 방식으로 사용자 경험을 설계해라. [청중] 결과물은 고등학생 학생회가 실제 사용할 수 있는 수준이어야 한다. 초보 사용자도 직관적으로 사용할 수 있어야 한다. [임무] 학교 온라인 건의함 MVP를 제작하라. [필수 조건] - 반드시 HTML/CSS/JS를 하나의 index.html 파일로 작성 - 서버 사용 금지 / localStorage 사용 가능 - 모바일 반응형 필수 / 접근성 고려 - 유지보수 가능한 구조 / 코드 가독성 / 주석 포함 - 자연스럽고 현대적인 UI 사용 [필수 기능] 1. 익명 건의 작성 2. 건의 목록 출력 3. 좋아요 기능 4. 검색 기능 5. 삭제 기능 6. 다크모드 7. 입력 유효성 검사 8. 빈 상태 UI 9. 모바일 UI 최적화 [좋은 UI 예시] 카드형 레이아웃 / hover 효과 / 명확한 버튼 구분 / 적절한 여백 / 모바일 터치 최적화 [좋지 않은 예시] 브라우저 기본 스타일 / 정렬되지 않은 레이아웃 / 작은 버튼 / 기능이 동작하지 않는 UI [출력 형식] 1. 설계 설명 2. 기능 구조 설명 3. 전체 코드 4. 개선 포인트 5. 예상 한계점 [중요] 절대로 기능 누락 금지. 절대로 placeholder 수준 UI 금지. 절대로 설명 없는 코드 출력 금지. [질문] 정보가 부족하면 먼저 질문해라. 충분한 정보를 얻기 전까지 구현을 시작하지 마라.
Answer C 구현을 시작하기 전에 실제 사용 환경을 명확히 해야 완성도 높은 MVP를 만들 수 있습니다.현재 요구사항만으로도 기본 구현은 가능하지만, 아래 6가지가 결정되면 UX와 구조 품질이 크게 좋아집니다. 1. 학생들이 건의를 “카테고리”로 분류해야 하나요? 예: * 시설 * 급식 * 수업 * 행사 * 기타 → 있으면 검색성과 관리성이 좋아집니다. 2. 건의 상태 표시가 필요한가요? 예: * 접수됨 * 검토중 * 해결완료 → 학생회 운영용 MVP라면 실제로 매우 유용합니다. 3. 삭제 기능은 누구나 삭제 가능한 구조인가요? 현재 서버 없이 localStorage만 사용하므로 선택지가 있습니다: * A. 작성 후 즉시 삭제 가능 * B. 비밀번호 입력 후 삭제 * C. 관리자 모드 없음 → 모두 삭제 가능 * D. 관리자 PIN 방식 → UX와 보안 수준이 달라집니다. 4. 첨부 이미지 기능이 필요한가요? 예: * 급식 사진 * 시설 고장 사진 → localStorage 기반에서도 Base64로 간단 MVP 구현 가능하지만 용량 제한이 생깁니다. 5. 학생회용 “통계 UI”가 필요하나요? 예: * 총 건의 수 * 오늘 작성 수 * 인기 건의 TOP3 → UX 완성도가 크게 올라갑니다. 6. 디자인 톤은 어느 방향이 좋나요? 선택 예시: * A. 깔끔한 학교 공지 느낌 * B. 인스타/커뮤니티 느낌 * C. Notion 스타일 * D. Discord 스타일 다크 UI * E. 애플 스타일 미니멀 UI 답변해주시면 그에 맞춰: * 실제 서비스 수준 구조 설계 * 모바일 UX * 접근성 * 유지보수 가능한 단일 index.html * 현대적 UI * 완전 동작형 코드 까지 포함해서 제작하겠습니다.
PROMPT C 1. 학생들의 건의를 '카테고리'로 분류해라. 2. 건의 상태 표시가 필요하다. 3. 현재 MVP에서의 삭제 기능은 B로 한다. 4. 이미지 첨부 기능이 필요하다. 5. 학생회용 '통계 UI'가 필요하다. 6. 디자인 톤은 A와 같은 스타일로 구현한다.
Answer C 1. 설계 설명 이번 MVP는 실제 고등학교 학생회가 사용할 수 있는 수준을 목표로 설계되었다.단순 CRUD 예제가 아니라, 다음과 같은 “실사용 흐름” 중심 UX를 기준으로 구성했다. 핵심 UX 목표 * 학생은 빠르게 익명 건의를 작성 가능 * 모바일에서도 한 손 조작 가능 * 학생회는 건의 상태를 쉽게 관리 가능 * 빈 화면에서도 어색하지 않은 UI * 초보 사용자도 학습 없이 사용 가능 전체 구조 단일 파일 구조 (index.html) 하나의 index.html 안에서: * HTML → 구조 * CSS → 디자인 시스템 * JavaScript → 상태 관리 및 기능 처리 를 모두 포함했다. 데이터 저장 방식 서버 없이 동작해야 하므로: * localStorage * 이미지 Base64 저장 방식을 사용했다. 저장 데이터 예시: { id, title, content, category, status, likes, password, image, createdAt } UI 설계 방향 디자인 톤: * 학교 공지 시스템 * 깔끔한 카드형 레이아웃 * 차분한 블루 포인트 컬러 * 과한 애니메이션 제거 * 접근성 대비 색상 사용 접근성 고려 사항 * 버튼 aria-label 적용 * 충분한 터치 영역 * 키보드 포커스 지원 * 색상 대비 확보 * 모바일 폰트 가독성 확보 2. 기능 구조 설명 1. 익명 건의 작성 기능: * 제목 * 내용 * 카테고리 * 삭제 비밀번호 * 이미지 첨부 유효성 검사: * 최소 글자 수 * 공백 입력 방지 * 비밀번호 길이 검사 2. 건의 목록 출력 카드형 UI로 출력: * 제목 * 카테고리 * 상태 * 작성일 * 좋아요 수 * 첨부 이미지 3. 좋아요 기능 * 건의별 좋아요 증가 * localStorage 자동 저장 4. 검색 기능 검색 대상: * 제목 * 내용 * 카테고리 실시간 필터링 지원 5. 삭제 기능 비밀번호 기반 삭제: * 삭제 버튼 클릭 * 비밀번호 입력 * 일치 시 삭제 6. 다크모드 * localStorage 저장 * 시스템 느낌 유지 * 명암 대비 유지 7. 상태 관리 기능 학생회 관리 목적: * 접수됨 * 검토중 * 해결완료 드롭다운 변경 가능 8. 통계 UI 표시: * 전체 건의 수 * 해결 완료 수 * 총 좋아요 수 9. 빈 상태 UI 게시글 없을 때: * 안내 메시지 * 자연스러운 아이콘 * UX 공백 제거 3. 전체 코드 ... (코드 생략) 4. 개선 포인트 1. 관리자 인증 시스템 현재: * 누구나 상태 변경 가능 개선: * 학생회 관리자 PIN * 관리자 로그인 2. 이미지 최적화 현재: * Base64 저장 문제: * localStorage 용량 제한 개선: * 이미지 압축 * IndexedDB 사용 3. 정렬 기능 추가 가능: * 최신순 * 좋아요순 * 해결완료순 4. 알림 UX 현재: * alert 사용 개선: * Toast UI * 모달 UI 5. PWA 지원 가능: * 모바일 앱처럼 설치 가능 5. 예상 한계점 1. localStorage 용량 제한 이미지 첨부가 많아지면 저장 한계 발생. 2. 서버 미사용 구조 현재: * 같은 브라우저에서만 데이터 유지 즉: * 학교 전체 공유 불가 3. 보안 한계 현재 비밀번호: * 클라이언트 저장 실서비스에서는: * 암호화 필요 4. 관리자 기능 제한 현재: * 관리자 권한 분리 없음 5. 동시 사용 불가능 여러 학생이 동시에 공유 사용 불가능. 이 MVP는: * 실제 발표용 * 프론트엔드 포트폴리오 * 학생회 프로토타입 * UX 연구용 으로 충분한 수준의 구조와 완성도를 가진다.
